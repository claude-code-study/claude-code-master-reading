## 개발자의 핵심 역량 (p34)

- 기본적인 개발 언어 문법을 이해한 상태에서 빠르게 요구사항을 분석하고, AI가 정확하게 작업할 수 있도록 업무를 분배하며, 출력물을 검토하고, 테스트의 품질과 정확성을 점진적으로 보정하는 능력

### 좋은 개발자

- **과거**: 문제를 해결하고 빠르고 정확하게 코드를 작성하는 사람
- **지금**:
    - AI 도구를 통해 여러 작업을 지시하고 조율하여 문제를 즉각적으로 수정하는 역할Orchestrator
    - 개발자가 직접 코드를 작성하는 행위 자체가 오히려 병목

### AI 시대에 맞춰 새롭게 기대되는 개발자의 역량

- 설계자Architect: 무엇을 만들 것인지, 어떤 구조로 만들 것인지 결정
- 검토자Reviewer: AI가 만든 코드의 품질, 보안, 성능을 판단
- 의사결정자Decision Maker: 여러 대안 중 최선을 선택
- 오케스트레이터Orchestrator: AI와의 협업을 조율

### 변하지 않는 것

- 문제를 해결해나가는 능력
- 시시각각 변하는 비즈니스 도메인에 대한 이해
- 법적, 윤리적 이슈에 대한 판단력
- 부서 간, 역할 간 커뮤니케이션 능력
- AI 도구와의 커뮤니케이션 능력과 도구의 활용 능력

## AI 코딩 도구의 발전 단계 및 특징 (p35~37)

**1. 초기 탐색 단계 (2022년 말 ~ 2023년 초)**

- **주요 도구:** ChatGPT (GPT-3.5/4)
- **특징:** IDE(개발 도구)와 분리된 상태에서 웹 브라우저를 통해 활용.
- **용도:** 특정 알고리즘 구현, 테스트 코드 작성, 간단한 리팩터링 등 **단편적인 코드 생성** 및 보조 용도.
- **한계:** 프로젝트 전체의 `맥락(Context) 파악이 어려워 복사-붙여넣기 방식에 의존`함.

**2. AI 코딩 어시스턴트 단계 (2023년 ~ 현재)**

> "AI가 제안하고, 개발자가 검토·수정하는 구조"
> 
- 주요 도구: **GitHub Copilot, Cursor(초기), Windsurf, JetBrains AI Assistant**
- 특징: IDE와 직접 통합되어 프로젝트 내 파일들의 맥락을 실시간으로 파악.
- 주요 기능: * Tab 완성: 다음 코드를 예측하여 제안.
    - 인라인 편집: 자연어로 명령하면 해당 위치의 코드를 즉시 수정.
- 역할 분담: 개발자가 코드 작성의 주체이며, `AI는 오타를 줄이고 작성 속도를 높여주는 조수` 역할.

**3. AI 에이전트 단계 (2024년 말 ~ 현재 급부상)**

> "목표를 부여하면 AI가 스스로 계획하고 실행하는 구조"
> 
- 주요 도구: **Claude Code, Cursor (Agent 모드), GitHub Copilot Workspace, Devin.**
- 특징: `단순히 코드를 제안하는 것을 넘어, 문제 해결을 위한 '자율성'을 가짐`
- 주요 기능:
    - `자율적 실행`: "로그인 기능을 만들어줘"라고 하면 스스로 계획 수립 -> 관련 파일 탐색 -> 코드 작성 -> 테스트 실행 -> 오류 수정 과정을 반복.
    - 도구 사용: 터미널 명령 실행, 파일 생성/삭제, 브라우저 테스트 등을 직접 수행.
- 역할 분담: 개발자는 목표(Goal)를 설정하고 결과물을 최종 승인하는 감독관 역할, AI는 실행 주체.
- 강점:
    - 맥락 전환 비용을 줄여준다
- **한계:**
    - AI가 생성하는 코드는 항상 정확하지 않다
    - 에이전트 도구는 코드를 직접 파일에 쓰기 때문에 검토없이 진행하면 코드베이스에 바로 반영된다.
        - ex) 문법적으로 올바르지만 논리적으로 틀린 코드, 존재하지 않는 API를 호출하는 코드, 의도와 다른 방향으로 구현된 코드가 프로젝트에 섞여 들어갈 수 있다.
    - 컨텍스트 윈도우의 제한. 대규모 프로젝트에서 모든 관련 파일을 컨텍스트에 넣을 수 없기 때문에 AI가 프로젝트의 전체 구조나 기존 패턴을 파악하지 못해 일관성 없는 코드를 생성할 수 있다
    - ⇒ 이러한 한계가 있기에, 작업 단위를 나누고, 어떤 정보를 먼저 전달할지, AI가 방향을 잘못 잡았을때 어떻게 교정할지가 중요하다.

## LLM 기반의 작동 원리

- LLM은 방대한 양의 텍스트 데이터, 특히 공개된 소스 코드와 기술 문서를 학습해 언어 패턴을 익힌다. 이 과정에서 프로그래밍 언어의 문법, 라이브러리 사용법, 일반적인 문제 해결 패턴을 내재화한다. LLM은 단순히 코드를 암기하는 것이 아니라 패턴을 학습해 새로운 상황에 적용한다.
- 사용자가 프롬프트를 입력하면 LLM은 학습된 패턴을 바탕으로 가장 적절한 다음 토큰Token을 순차적으로 예측한다.
- 컨텍스트 윈도우는 LLM이 한 번에 처리할 수 있는 텍스트 양을 말한다. 현재 프로젝트의 관련 파일, 대화 히스토리, 사용자 요청이 모두 이 윈도우 안에 들어간다. AI 코딩 어시스턴트는 에디터에서 열린 파일이나 커서 주변의 코드를 자동으로 수집해 LLM에 전달한다. 컨택스트 윈도우가 커질수록 토큰의 처리량이 비례하여 늘어나며, 양이 늘수록 오류의 확률도 높아진다.

## 클로드 코드

- 앤트로픽Anthropic이 2025년 2월 출시한 CLI 기반 AI 에이전트
    - `CLI 환경`을 선택한 이유? 파일 조작, 명령어 실행, 다중 파일 작업 같은 에이전틱 동작을 수행하는데 CLI가 훨씬 자연스럽고, `IDE에 종속되지 않아 어떤 개발 환경에서든 동일하게 작동`하기 때문. 새로운 인터페이스를 배울 필요가 없다.
    - IDE를 설치하고 실행하는건 무겁지만 터미널만 있으면 되니 가볍다.
    - 터미널 기반이기 때문에 기존 개발 도구들과 충돌 없이 함께 쓸수 있어 통합의 유연성이 있다.
- 터미널에서 자연어로 대화하면서 파일 생성, 코드 작성, 명령어 실행까지 AI가 직접 수행한다.

### 특징

1. 터미널 네이티브
    1. 처음부터 터미널 환경을 위해 설계되었고, 터미널에서 독립적으로 실행된다. → 자동화가 쉽다.
2. CLAUDE.md를 통한 프로젝트 컨텍스트
    1. 프로젝트 루트에 있는 [CLAUDE.md](http://CLAUDE.md) 파일을 읽어 프로젝트의 규칙과 컨텍스트를 파악한다. 코딩 컨벤션, 아키텍처 원칙, 자주 사용하는 패턴 등을 이 파일에 정리해두면 클로드 코드가 프로젝트에 맞는 코드를 생성한다.
3. 에이전트 방식의 자율적 흐름
    1. 클로드 코드는 계획을 세우고 실행하고 검증하는 흐름을 스스로 진행한다.
4. 확장성 - Hook, MCP, Plugin
    1. 다양한 방식으로 기능을 확장할 수 있다. **`Hook을 설정하면` 특정 이벤트가 발생할 때 자동으로 스크립트를 실행할 수 있다. ex) 작업이 완료되면 알림 소리를 낸다.**
    2. MCP Model Context Protocol을 통해 외부 도구와 연동할 수 있다.
    3. **플러그인 시스템을 활용하면 팀 고유의 워크플로를 패키징해 공유할 수도 있다.**

## 더 많은 실험이 가능해진 시대

아이디어만 있다면 기능을 정의하는 몇 줄의 문장과 기술 스택 정도만으로 한 시간 안에 동작하는 애플리케이션을 만들어낼 수 있다.

‘먼저 만들고, 검증하고, 개선하는’ 프로토타입 우선 접근법을 가능하게 만든다. 몇 시간 만에 동작하는 프로토타입을 만들 수 있다는 건 기획 단계에서 파워포인트 대신 실제 동작하는 목업Mock-up으로 이해관계자와 소통할 수 있다는 의미

실패의 비용이 달라졌다. 과거의 실패는 무거웠으나, 현재의 실패는 몇 시간 투자로 판단이 가능할 정도로 가벼운 노력만 기울이면 된다.

주의할 점은 AI로 빠르게 만든 코드의 유지보수 문제(바이브 코딩 숙취Vide coding hangover) ⇒ 테스트 주도 개발, 명세 작성, 코드 리뷰 같은 검증 체계가 필요하다.
